# 预发布审查清单

## 说明

审查 `git diff origin/main` 输出中列出的以下问题。要具体 — 引用 `file:line` 并建议修复。跳过任何没问题的内容。只标记真正的问题。

**两遍审查：**
- **第一遍（关键）：** 首先运行 SQL 和数据安全以及 LLM 输出信任边界。最高严重性。
- **第二遍（信息性）：** 运行所有剩余类别。较低严重性但仍需处理。

所有发现都通过修复优先审查进行处理：明显的机械修复会自动应用，
真正模糊的问题会批量处理成单个用户问题。

**输出格式：**

```
预发布审查：N 个问题（X 个关键，Y 个信息性）

**已自动修复：**
- [file:line] 问题 → 已应用修复

**需要输入：**
- [file:line] 问题描述
  推荐修复：建议的修复方案
```

如果未发现问题：`预发布审查：未发现问题。`

要简洁。对于每个问题：一行描述问题，一行描述修复。没有前言，没有总结，没有"总体看起来不错"。

## 审查类别

### 第一遍 — 关键

#### SQL 和数据安全
- SQL 中的字符串插值（即使值是 `.to_i`/`.to_f` — 使用 `sanitize_sql_array` 或 Arel）
- TOCTOU 竞争：应该是原子 `WHERE` + `update_all` 的检查然后设置模式
- `update_column`/`update_columns` 绕过对有或应该有约束的字段的验证
- N+1 查询：循环/视图中使用的关联缺少 `.includes()`（尤其是头像、附件）

#### 竞争条件和并发
- 没有唯一性约束或 `rescue RecordNotUnique; retry` 的读-检查-写（例如 `where(hash:).first` 然后 `save!` 没有处理并发插入）
- 在没有唯一数据库索引的列上使用 `find_or_create_by` — 并发调用可能创建重复项
- 不使用原子 `WHERE old_status = ? UPDATE SET new_status` 的状态转换 — 并发更新可能跳过或重复应用转换
- 对用户控制数据使用 `html_safe`（XSS）— 检查任何 `.html_safe`、`raw()` 或插入到 `html_safe` 输出的字符串插值

#### LLM 输出信任边界
- LLM 生成的值（电子邮件、URL、名称）写入数据库或传递给邮件程序而没有格式验证。在持久化之前添加轻量级保护（`EMAIL_REGEXP`、`URI.parse`、`.strip`）。
- 结构化工具输出（数组、哈希）在数据库写入前未经过类型/形状检查。

#### 枚举和值完整性
当差异引入新的枚举值、状态字符串、层级名称或类型常量时：
- **跟踪它通过每个消费者。** 阅读（不要只是 grep — 阅读）每个切换、过滤或显示该值的文件。如果任何消费者不处理新值，标记它。常见遗漏：向前端下拉菜单添加值但后端模型/计算方法不持久化它。
- **检查允许列表/过滤数组。** 搜索包含兄弟值的数组或 `%w[]` 列表（例如，如果向层级添加 "revise"，找到每个 `%w[quick lfg mega]` 并验证 "revise" 在需要的地方被包含）。
- **检查 `case`/`if-elsif` 链。** 如果现有代码基于枚举分支，新值是否会落入错误的默认值？
要做到这一点：使用 Grep 找到所有对兄弟值的引用（例如，grep "lfg" 或 "mega" 找到所有层级消费者）。阅读每个匹配项。此步骤需要阅读差异之外的代码。

### 第二遍 — 信息性

#### 条件副作用
- 代码路径在条件上分支但忘记在一个分支上应用副作用。例如：项目被提升为已验证，但仅当次要条件为真时才附加 URL — 另一个分支在没有 URL 的情况下提升，创建不一致的记录。
- 声称动作发生但动作被有条件跳过的日志消息。日志应反映实际发生的情况。

#### 魔术数字和字符串耦合
- 在多个文件中使用的裸露数字字面量 — 应该是一起记录的命名常量
- 错误消息字符串在其他地方用作查询过滤器（grep 字符串 — 是否有任何匹配？）

#### 死代码和一致性
- 分配但从未读取的变量
- PR 标题与 VERSION/CHANGELOG 文件之间的版本不匹配
- 描述更改不准确的 CHANGELOG 条目（例如，"从 X 更改为 Y" 当 X 从未存在时）
- 描述代码更改后旧行为的注释/文档字符串

#### LLM 提示问题
- 提示中的 0 索引列表（LLM 可靠地返回 1 索引）
- 列出可用工具/功能的提示文本与 `tool_classes`/`tools` 数组中实际连接的不匹配
- 在多个地方声明的可能漂移的单词/令牌限制

#### 测试差距
- 断言类型/状态但不是副作用的负路径测试（URL 附加？字段填充？回调触发？）
- 断言字符串内容而不检查格式（例如，断言标题存在但不是 URL 格式）
- 当代码路径应明确不调用外部服务时缺少 `.expects(:something).never`
- 安全强制执行功能（阻止、速率限制、认证）没有集成测试验证强制执行路径端到端工作

#### 完整性差距
- 完整版本将花费 <30 分钟 CC 时间的快捷实现（例如，部分枚举处理、不完整的错误路径、易于添加的缺失边缘情况）
- 仅显示人类团队努力估计的选项 — 应显示人类和 CC+gstack 时间
- 添加缺失测试是"湖泊"而非"海洋"的测试覆盖差距（例如，缺失的负路径测试、镜像愉快路径结构的缺失边缘情况测试）
- 功能实现达到 80-90%，而 100% 可通过适度的额外代码实现

#### 加密和熵
- 截断数据而不是哈希（最后 N 个字符而不是 SHA-256）— 更少的熵，更容易碰撞
- 用于安全敏感值的 `rand()` / `Random.rand` — 改用 `SecureRandom`
- 对机密或令牌的非恒定时间比较（`==`）— 易受计时攻击

#### 时间窗口安全
- 假设"今天"覆盖 24 小时的日期键查找 — 仅在 PT 上午 8 点报告只看到今天键下的午夜→上午 8 点
- 相关功能之间不匹配的时间窗口 — 一个使用小时桶，另一个对相同数据使用日键

#### 边界处的类型强制
- 类型可能改变的 Ruby→JSON→JS 边界交叉的值（数字 vs 字符串）— 哈希/摘要输入必须规范化类型
- 哈希/摘要输入在序列化前不调用 `.to_s` 或等效方法 — `{ cores: 8 }` vs `{ cores: "8" }` 产生不同的哈希

#### 视图/前端
- 部分中的内联 `<style>` 块（每次渲染都重新解析）
- 视图中的 O(n*m) 查找（循环中的 `Array#find` 而不是 `index_by` 哈希）
- 可以是 `WHERE` 子句的 DB 结果的 Ruby 端 `.select{}` 过滤（除非有意避免前导通配符 `LIKE`）

## 修复优先启发式

此启发式被 `/review` 和 `/ship` 引用。它确定
代理是否自动修复发现或询问用户。

```
自动修复（代理无需询问即可修复）：     询问（需要人工判断）：
├─ 死代码 / 未使用的变量            ├─ 安全（认证，XSS，注入）
├─ N+1 查询（缺少 .includes()）      ├─ 竞争条件
├─ 与代码矛盾的过时注释             ├─ 设计决策
├─ 魔术数字 → 命名常量             ├─ 大型修复（>20 行）
├─ 缺少 LLM 输出验证               ├─ 枚举完整性
├─ 版本/路径不匹配                 ├─ 移除功能
├─ 分配但从未读取的变量             └─ 任何改变用户可见
└─ 内联样式，O(n*m) 视图查找          行为
```

**经验法则：** 如果修复是机械的，并且高级工程师会在没有讨论的情况下应用它，那么它是 AUTO-FIX。如果合理的工程师可能对修复有不同意见，那么它是 ASK。

**关键发现默认倾向于 ASK**（它们本质上风险更高）。
**信息性发现默认倾向于 AUTO-FIX**（它们更机械）。

## 抑制 — 不要标记这些

- 当冗余无害且有助于可读性时，"X 与 Y 冗余"（例如，`present?` 与 `length > 20` 冗余）
- "添加注释解释为什么选择此阈值/常量" — 阈值在调优期间变化，注释会过时
- 当断言已经覆盖行为时，"此断言可以更严格"
- 建议仅一致性更改（将值包装在条件中以匹配另一个常量的保护方式）
- 当输入受到约束且 X 在实践中从未发生时，"Regex 不处理边缘情况 X"
- "测试同时练习多个保护" — 没关系，测试不需要隔离每个保护
- 评估阈值变化（max_actionable，最小分数）— 这些是凭经验调整的，并且不断变化
- 无害的无操作（例如，对永远不在数组中的元素使用 `.reject`）
- 您正在审查的差异中已经解决的任何问题 — 在评论前阅读完整差异
